## 1. Authentication

All requests to `https://porn-app.com/api/` require a `hash` header containing an RSA-encrypted device fingerprint.

### Hash Generation

The hash is generated by:
1. Creating a JSON payload with device information
2. RSA-encrypting it with the app's public key (4096-bit, PKCS1 padding)
3. Base64-encoding the result

**Hash payload structure:**
```json
{
  "id": "<android_id or random string>",
  "isTV": false,
  "loginStatus": {
    "pro": 0,
    "status": 0,
    "token": "",
    "unixtime": <timestamp>,
    "user_id": 0
  },
  "packageName": "com.streamdev.aiostreamer",
  "signatures": ["VQMyUhZdmnnwK5RVCbeGqu0HN020MEDUM44crQyL1zw="],
  "time": <timestamp>,
  "version": 6643
}
```

- `id`: Android device ID (can be any string for testing)
- `time`/`unixtime`: Current Unix timestamp
- `signatures`: SHA-256 hash of the APK's certificate DER bytes (base64-encoded)
- `version`: App version code (6643 = v6.6.4)

**Example hash generation script** (`gen_hash.sh`):
```bash
#!/bin/bash
TIMESTAMP=$(date +%s)
HASH_JSON=$(python3 -c "
import json, collections
ts = $TIMESTAMP
payload = collections.OrderedDict([
    ('id', 'a1b2c3d4e5f6a7b8'),
    ('isTV', False),
    ('loginStatus', collections.OrderedDict([
        ('pro', 0),
        ('status', 0),
        ('token', ''),
        ('unixtime', ts),
        ('user_id', 0)
    ])),
    ('packageName', 'com.streamdev.aiostreamer'),
    ('signatures', ['VQMyUhZdmnnwK5RVCbeGqu0HN020MEDUM44crQyL1zw=']),
    ('time', ts),
    ('version', 6643)
])
print(json.dumps(payload, separators=(',', ':')))
")
echo -n "$HASH_JSON" | openssl pkeyutl -encrypt -pubin -inkey pubkey.pem -pkeyopt rsa_padding_mode:pkcs1 | openssl base64 -A | tr -d '\n\r'
```

### Required Headers

```
hash: <RSA-encrypted hash string>
Content-Type: application/json
Authorization: Bearer <token>  # Optional for free content
```

**Note:** Invalid or missing hash returns: `{"error":"Server Error","message":"Application Error - Please redownload the app on www.porn-app.com !","status":500}`

### 2.5 Get Video Listings (VERIFIED WORKING — see §7.7 for server-side note)

**CRITICAL:** All POST bodies MUST be gzip-compressed with `Content-Encoding: gzip` header. The server's OkHttp interceptor (`s92.java`) automatically compresses all request bodies.

```bash
HASH=$(bash gen_hash.sh)

# The payload field is a JSON-encoded STRING (double-encoded).
# The outer JSON has {"payload":"<string>"}, where <string> is itself JSON.
echo -n '{"payload":"{\"viewer\":\"new\",\"page\":1,\"category\":false}"}' | gzip | \
curl -s -X POST "https://porn-app.com/api/v9/sites/beegcom/data?isTV=false" \
  -H "hash: $HASH" \
  -H "Authorization: Bearer " \
  -H "Content-Type: application/json; charset=UTF-8" \
  -H "Content-Encoding: gzip" \
  -H "User-Agent: okhttp/5.3.2" \
  --data-binary @-
```

**Payload format rules** (confirmed via server error messages):
- Empty string `""` → server uses default `newUrl` from `/info` config
- JSON string with `viewer`, `page` → server parses and uses appropriate URL
- Missing `viewer` field → `500: "Undefined array key 'viewer'"`
- Missing `page` field → `500: "Undefined array key 'page'"`
- Non-string payload (object/number) → `500: "Cannot access offset of type string on string"`

StandardFilter payload (most sites):
```json
{"viewer":"new","page":1,"category":false}
```

Viewer options: `new` (newest), `hot` (hottest), `views` (most viewed)

With category:
```json
{"viewer":"new","page":1,"category":true,"tag":"amateur"}
```

With search:
```json
{"viewer":"new","page":1,"category":false,"search":"query here"}
```

## 4. Client-Side Scraping (Direct, No Backend)

These sites have dedicated resolver classes in the app that bypass the porn-app.com backend entirely:

### 4.1 ePorner

1. GET video page, extract `EP.video.player.vid` and `EP.video.player.hash`
2. Compute hash: split 32-char hex into 4x8-char segments, convert each `BigInteger(hex,16).toString(36)`, concatenate
3. GET `https://www.eporner.com/xhr/video/{vid}?hash={computed}&domain=www.eporner.com&supportedFormats=mp4`

### 4.2 Javtiful

1. GET video page, extract CSRF token from `[data-csrf-token]` and video ID from `vcpov` JS var
2. POST `https://javtiful.com/ajax/get_cdn` with `video_id`, `token`
3. Read CDN URL from `X-Message` response **header** (not body)

### 4.3 Sexu

POST `https://sexu.com/api/video-info` with `videoId={id}` (multipart form)

### 4.4 SexTB

1. Load page via HtmlUnit (headless browser, JS disabled)
2. Find `.btn-player` elements, extract `data-source` and `data-id`
3. POST `https://sextb.net/ajax/player` with `episode={id}&filmId={source}` + Authorization header from response

## 6. Typical Flow

```
1. GET  /v9/sites                     → get all 217 sites
2. POST /v9/sites/{tag}/info          → get scraping config (CSS selectors, URL templates)
3. POST /v9/sites/{tag}/data          → get video listings
4. POST /v9/sites/{tag}/stream        → resolve video to m3u8 URLs
5. POST /v9/videoheaders              → get Referer/Origin headers for playback
6. Play m3u8 with ExoPlayer + custom headers
```

For sites with dedicated resolvers (ePorner, Javtiful, Sexu, SexTB), steps 3-5 are replaced by direct scraping.

No user account is required for browsing free content. The `Authorization: Bearer` header is only needed for cloud sync (favorites, history, playlists).

## 7. Real Traffic Analysis (Appetize Sandbox Capture)

The following was captured from the actual app running on an Android 16 emulator via Appetize.io.

### 7.1 Startup Sequence

The app follows this exact order on launch:

```
1. GET  /                          → WebView landing page
2. GET  /css/nav.css, /css/adult.css, /js/ad_slider.js, etc.  → static assets
3. GET  /videos/brazzers5.mp4      → background video (Range: bytes=0-)
4. GET  /download                  → APK download (chunked: Range: bytes=0-, 7491482-, 14982964-)
5. GET  /api/v9/checkInfo          → device/country check (hash header)
6. GET  /api/update                → app update check (no hash)
7. POST /api/v9/device             → device registration (gzip body)
8. GET  /news                      → WebView news page (spoofed iOS UA)
9. POST /api/v9/login              → session auth (gzip body, repeated many times)
10. POST /api/v9/unixTime          → time sync (gzip body)
11. GET  /api/update               → second update check
12. GET  /api/v9/sites             → full site list (hash header)
13. GET  /imgs/sites/*.png         → site icons (Dalvik UA)
```

### 7.2 Per-Site Request Flow

When a user taps a site (e.g., `beegcom`):

```
1. POST /api/v9/sites/beegcom/info       → scraping config
2. GET  /api/v9/categories?site=beegcom  → category list
3. POST /api/v9/login                    → session refresh
4. POST /api/v9/sites/beegcom/link       → resolve stream URLs
5. POST /api/v9/sites/beegcom/data?isTV=false  → video listings
6. GET  /nat1/beegcom/new                → WebView proxy (if needed)
```

**Login is polled repeatedly** — called before almost every data/link request as a session keep-alive.

### 7.3 Request Body Compression

All POST requests from OkHttp include `Content-Encoding: gzip`. The app's `s92` OkHttp interceptor automatically gzip-compresses request bodies before sending. This is critical for:

- `/api/v9/device` (form-urlencoded body, gzipped)
- `/api/v9/unixTime` (form-urlencoded body, gzipped)
- `/api/v9/login` (JSON body, gzipped)
- `/api/v9/sites/{tag}/data` (JSON body, gzipped)
- `/api/v9/sites/{tag}/link` (JSON body, gzipped)

**Note:** GET endpoints (`/v9/sites`, `/v9/categories`, `/v9/checkInfo`, `/v9/starter`) work fine without gzip. POST endpoints with bodies may return errors or empty results without proper gzip encoding.

### 7.5 WebView Proxy Routes (`/nat1/`, `/nat2/`)

When the app cannot extract a direct video URL, it loads the site through the server-side proxy:

```
GET /nat1/{sitetag}/new     (e.g., /nat1/beegcom/new, /nat1/xxxtube/new)
GET /nat2/{sitetag}/new     (fallback route, e.g., /nat2/peekvidscom/new)

Headers required:
```

These return full HTML pages that the app renders in an embedded WebView. The server acts as a proxy, fetching and re-rendering the third-party site's content.

**Confirmed findings (June 2026):**
- Hash authentication is VALID — `/v9/unixTime` returns `{"statusCode":200,"result":"OKEY"}`
- All config endpoints work: `/v9/sites`, `/v9/checkInfo`, `/v9/starter`, `/v9/sites/{tag}/info`, `/v9/categories`
- Login works: returns `{"status":404,"pro":0,"unixtime":...,"token":""}` for unauthenticated users
- Device registration works: returns `{"statusCode":200,"result":""}`
- The app works for users in India (confirmed by user)

**`/data` endpoint behavior:**
The `/data` endpoint currently returns `404: "Could not fetch website content! Website might be down!"` for all tested sites (35+ sites across all categories). This is a **server-side scraping infrastructure issue** — the server cannot fetch the external URLs (e.g., `store.externulls.com` for beegcom) that it needs to scrape video listings.

This is NOT caused by:
- Invalid hash (hash works, other endpoints succeed)
- Missing gzip (tested with and without, same 404)
- Geo-restriction (user confirmed app works in India)
- User-Agent differences (tested okhttp, Chrome, Dalvik, desktop — all 404)
- Wrong payload format (empty string triggers the fetch attempt, then fails)

The `/link` endpoint also returns `500: "Unexpected error occured"` — further confirming the server's scraping backend is having issues.

**Nat proxy routes (`/nat1/`, `/nat2/`)** return a minimal ad-container HTML page (~2KB) with a JuicyAds/magsrv ad script but no actual video content. These are WebView wrapper pages for displaying ads.

### 7.8 Session Cookies

The server uses Laravel-style encrypted cookies:
- `XSRF-TOKEN` — CSRF protection (base64-encoded JSON with `{"iv":...}` format)
- `pornapp_session` — Laravel session (AES-256-CBC encrypted)
- `_ga`, `_gid`, `_gat_gtag_UA_25883073_2` — Google Analytics

These cookies are required for WebView proxy routes (`/nat1/`, `/nat2/`) but NOT for native API endpoints.
